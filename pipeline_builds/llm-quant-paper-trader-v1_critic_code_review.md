## BLOCK 🚫

### BUG: Regime filter primary path silently broken — wrong column name

**File:** `backtesting/live/lgbm_signal_generator.py`, method `_get_regime()`

**The problem:**

```python
# Current (WRONG):
if "ema_cross_9_21" in df.columns:
    val = last["ema_cross_9_21"].to_numpy()[0]
    if val == 0:
        return "bear"
    elif val == 1:
        return "bull"
```

The column generated by `compute_ema_crossovers()` in `features.py` is **`ema_cross_9_21_signal`**, not `ema_cross_9_21`. So `"ema_cross_9_21" in df.columns` is always False → the primary path is skipped every time.

This is verifiable:
```python
# features.py line 418:
.alias(f"ema_cross_{fast}_{slow}_signal")  # → "ema_cross_9_21_signal"
```

**Secondary issue in the same block:** Even if the column name were fixed, the value interpretation is wrong. The code comment says "1=bullish, 0=bearish" but `ema_cross_9_21_signal` is a **crossover event** signal with values {1, -1, 0}, where:
- 1 = EMA 9 just crossed ABOVE EMA 21 (bullish crossover)
- -1 = EMA 9 just crossed BELOW EMA 21 (bearish crossover)  
- **0 = no crossover this bar** (the vast majority of bars)

So `val == 0: return "bear"` would flag nearly every bar as "bear". The column being looked for isn't a persistent regime state — it's an event.

**The correct column to use is `ema_cross_9_21_dist`:** normalized distance `(ema_9 - ema_21) / ema_21` — this IS a persistent state (negative = bearish alignment, positive = bullish alignment).

**Current impact:** The `_get_regime()` always falls through to the ADX fallback. The ADX fallback (`adx > 25 and close < ema_9`) does work, but uses ema_9 (9-bar EMA) as the threshold instead of the 9/21 cross — this is much noisier and produces different regime classifications.

**Fix:**

```python
# In _get_regime(), replace the first check:
if "ema_cross_9_21" in df.columns:
    val = last["ema_cross_9_21"].to_numpy()[0]
    if val == 0:
        return "bear"
    elif val == 1:
        return "bull"

# With:
if "ema_cross_9_21_dist" in df.columns:
    val = float(last["ema_cross_9_21_dist"].to_numpy()[0])
    if np.isnan(val):
        pass  # fall through to ADX
    elif val < 0:
        return "bear"
    elif val > 0:
        return "bull"
```

`ema_cross_9_21_dist` = `(ema_9 - ema_21) / ema_21`:
- Negative → EMA_9 below EMA_21 → bearish alignment → return "bear"
- Positive → EMA_9 above EMA_21 → bullish alignment → return "bull"
- Exactly 0 / NaN → neutral, fall through to ADX check

The ADX fallback can stay as-is as a secondary check.

## FLAG ⚑ (Medium)

### Confidence threshold / min order size interaction

**File:** `backtesting/live/paper_trader.py`

With `confidence_threshold=0.30` and default `base_size=$25`:
- A signal with confidence=0.31 → `size_usd = 25 × 0.31 = $7.75`
- `DydxExecutor.market_order()` will raise `DydxRiskError: order notional $7.75 is below minimum $10.00`
- This gets caught by the outer `except Exception` in `_process_symbol()` and logged as `"status": "error"` — confusing

**Suggestion (not blocking):** Add a pre-check in `_execute_signal` or `_process_symbol`:
```python
MIN_ORDER_USD = 10.0  # matches DydxExecutor.MIN_ORDER_NOTIONAL
if size_usd < MIN_ORDER_USD:
    log.info(f"{symbol}: size_usd=${size_usd:.2f} below minimum — skipping order")
    return None
```
Or document in README: "Set `--confidence-threshold` ≥ 0.40 when using default `--size 25` to ensure orders meet the $10 minimum (`$25 × 0.40 = $10`)."

This is a FLAG not a BLOCK because: (1) the error is caught safely, (2) paper trading stakes are low, (3) the default threshold of 0.30 is set intentionally for signal sensitivity.
