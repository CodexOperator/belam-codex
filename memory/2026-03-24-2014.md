What a proper "download and run" container would need 
 
 To make a fully self-contained image you could pull and run anywhere with one command, you'd need to solve these things the current Dockerfiles don't handle: 
 
 1. Everything inside the image, not bind-mounted 
 - Bake the workspace (belam-codex repo) into the image at build time via COPY or git clone 
 - Bake openclaw.json config into the image (or accept it via env vars / secrets at runtime) 
 - The current compose bind-mounts ~/.openclaw, ~/.ssh, and the workspace from the host — that defeats portability 
 
 2. All services in one container (or a proper compose stack) 
 - Gateway (openclaw) 
 - codex-render.py as a second process (use supervisord, s6-overlay, or a simple entrypoint script that launches both)
 - Cron jobs: either a cron daemon inside the container, or convert them to a sidecar container / scheduled tasks in the entrypoint

 3. Secrets management
 - SSH keys and API tokens can't be baked into the image (security risk). Pass them via: Docker secrets, env vars at runtime, or a mounted .env file
 - openclaw.json likely has credentials — same treatment

 4. Persistent state
 - Named volumes for: delivery-queue, agents, memory, logs, and anything that needs to survive a re-image
 - The workspace itself should be either in the image (immutable) or a volume (mutable)

 5. The re-image script would be straightforward:

 #!/bin/bash
 # rebuild.sh — one-command re-image
 set -e
 cd /path/to/belam-codex/docker

 # Pull latest code
 git pull origin main

 # Rebuild image with latest code baked in
 docker compose build --no-cache

 # Rolling restart (zero-downtime if healthcheck passes)
 docker compose up -d --force-recreate

 # Cleanup old images
 docker image prune -f

 echo "Done. New image running."
